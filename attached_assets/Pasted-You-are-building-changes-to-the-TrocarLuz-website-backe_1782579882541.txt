STRATEGIC OBJECTIVE
Turn TrocarLuz from an educational lead-capture site into a public residential GD (geração distribuída) comparison + switch-request engine, with the SEO/authority layer expanded in parallel.

POSITIONING (critical)
TrocarLuz means "compare and choose the best way to reduce your electricity bill" — NOT "change supplier" and NOT "trocar distribuidora."
For residential GD, NEVER imply the customer changes their distributor. Use:
"Você continua conectado à sua distribuidora, mas pode receber créditos/descontos na conta por meio de geração distribuída, dependendo da disponibilidade na sua região."
TrocarLuz ORCHESTRATES and ROUTES; the GD PARTNER executes the enrolment. Do not build contract automation.
Primary CTA: "Ver desconto disponível" · Secondary CTA: "Enviar conta de luz"

BUILD IN TWO PHASES — do not build everything at once.

=== COMPLIANCE & DATA (applies to all phases — do not skip) ===

WhatsApp:
- This is OPT-IN, inbound (the user requests messages on the site). Use the OFFICIAL WhatsApp Business Cloud API only — no unofficial automation.
- The proposal-ready message is sent outside the 24-hour window, so it MUST be a pre-approved UTILITY-category template. Submit all proactive messages as approved templates.
- Implement an inbound webhook to capture the customer's SIM/NÃO reply and map it to the switch request.
- This WhatsApp number/Business Manager is a CLEAN, DEDICATED consumer channel. It must never be used for, or linked to, cold outbound outreach. Keep it isolated.

LGPD / data protection:
- Two separate consent checkboxes at switch-request submission, both required:
  1. "Aceito receber mensagens da TrocarLuz/Ótima sobre minha análise e proposta pelo WhatsApp."
  2. "Autorizo o compartilhamento dos meus dados e da minha conta de luz com parceiros para a geração de uma proposta de economia."
- Link to a privacy policy. Store a data-retention rule.
- Bill files must be stored securely with signed/access-controlled URLs — never publicly accessible. Validate file type and size on upload.

Consumer protection (CDC):
- Estimates must be clearly indicative, never guaranteed.
- The 48h proposal validity must be framed as revalidation tied to real partner conditions, never fabricated scarcity. Use:
  "A proposta fica válida por 48 horas porque disponibilidade, desconto e condições do parceiro podem mudar. Depois desse prazo, será necessário revalidar."

PHASE 1 — ENGINEERING (ship fast)
============================================================
Goal: a working capture → estimate → bill → WhatsApp → routed switch request. Partner proposal return can be semi-manual at first (ops receives partner proposal and triggers the customer message); do NOT build the full partner-submission machine yet.

Customer flow:
1. User enters CEP, estado, distribuidora, valor médio da conta, tipo de imóvel, tipo (residência/empresa), possui carro elétrico? (sim/não).
2. User sees an INDICATIVE savings range with disclaimer.
3. User enters WhatsApp (mandatory) + both consent checkboxes.
4. User uploads electricity bill (secure storage).
5. User submits switch request → saved to DB → event SWITCH_REQUEST_SUBMITTED → integration hook fired → routed to the matching GD partner (by region) → confirmation WhatsApp template sent.

Estimate logic (honest):
- Rule-based, region-aware, clearly non-binding. Show a realistic GD discount range (e.g. typical residential GD savings band) for the region, NOT a fabricated precise number. Disclaimer:
  "Estimativas são indicativas e dependem da análise da conta, distribuidora, região, disponibilidade de parceiro e aprovação final."

Partner matching:
- Create a `partners` table/config keyed by state/distribuidora coverage. For MVP a lookup table (or single partner) is fine. If no partner covers the region → graceful "not available yet" path → capture to waitlist + offer content. Never strand the user.
